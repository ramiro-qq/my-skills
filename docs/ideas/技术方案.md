创建一个用于将原始需求转换为可开发技术架构方案的Skill:
1. 触发方式：生成技术方案、新功能、新需求等用户
2. 名词解释
- 新需求：如果用户未指定本次任务的原始需求，就检索项目中的MD文件，判断是需求文档的MD文件，并输出让用户确认需求，也可以让用户指定需求文档或简要描述需求， - 存量技术方案：默认将每次生成的技术方案存存到项目/docs/architectures目录下，文件名为：YYYY-MM-DD-[版本].md，版本号自动累加，如：v1、v2。新需求技术方案之前检索/docs/architectures目录下的MD方案文件，并基于最近(时间最晚，版本号最大)的文件作为现状存量技术方案
- 增量方案：本次新需求相较于存量方案和项目现状代码额外新增的业务功能，就是本次需要的增量方案
3. 其他能力
3.1 前端项目初始化能力
如果前端项目未初始化，就把项目初始化任务添加到技术方案设计中，并参考以下技术栈要求
SSR: NextJs + Shadcn/ui
SPA: vite@plus + react-router + typescript + shadcn/ui
3.2 技术方案调研能力
针对新需求，提供搜索市面主流相关技术方案搜索和对比能力，至少搜索选择3个解决方案，并结合本次新需求与项目要求，选择最适合的一个技术栈并说明原因
3.3 UI设计稿识别能力
如果指定了UI设计稿文件，需要对UI设计稿进行分析，提取全局公共的主题变量、设计公共组件、样式实现等
4. 技术方案模板
- 板块1: 技术可行性分析
基于增量需求，调研市场主流解决方案和技术选型，评估技术可行性，以及大致实现方式
- 板块2: 技术选型
结合增量需求 + 基础方案（最晚的存量技术方案）+ 项目架构要求（项目规范、约束、上下文要求等），调研市面上主流的技术选型，至少搜集3个同类型的技术实现案例。详细对比这三个技术在实现上的对比，主要包含：学习成本、集成难度、技术生态、维护状态、热门层度、改造难度等维度。
最终选择最适合当前项目的技术框架或者工具。
- 板块3: 架构设计
基于基础方案+增量需求+技术员选型+项目现状，分析设计项目的架构方案，方案要求：
要求1: 架构合理：保持良好的可扩展性和性能的前提下，能够实现新需求功能目标
要求2: 改造方案：基于当前项目现状和架构设计，整理出改动清单，描述清楚影响范围
要求3:模块拆分：基于业务形态和功能，在保证合理的情况下拆分出功能模块
- 后端服务可以基于模块拆分，归类到不同的模块中，便于服务拆分、接口路径归类
- 前端项目可以基于模块拆分，归类到不同的模块目录中，并抽离公共组件、权限控制、接口定义等公共逻辑处理，提升复用性
要求4: 公共逻辑、公共组件提取：方案必须基于当前项目的技术规范和选型，尽量抽离可复用、灵活性更高的公共组件、方法、Hooks、状态管理和全局事件等，列出清单。
要求5: 影响范围：基于前面的技术方案设计，评估影响范围，整理功能回归清单
- 板块3.1：架构图
基于架构设计，生成项目架构图
- 板块3.2：流程示意图
基于架构设计，生成本项目关于数据扭转、数据走向等的流程示意图
- 版本3.3：重难点功能详细技术设计
针对架构设计和新需求中的功能清单，选择重要或者是难度较大的功能，并将每个功能的目标、设计思路、实现方式、代码示例（如有必要）、组件提取、公共逻辑代码等进行说明，不要太详细，需要让人看明白是思路和实现。
- 板块4: 影响范围
结合新需求与架构设计，会对项目现状产生哪些影响，包括但不限于：组件改造、新增文件、现有代码改动等，总结并列举有变化的地方，便于后期设计测试用例

执行步骤：
步骤一：与用户确认新需求，可以是指定或者扫描出来的需求文档，也可以是用户的输入。可以让用户确定UI设计稿位置或者连接（可选）
步骤二：基于新需求完成技术方案的调研、选型和UI设计稿的解析
步骤三：根据新需求、存量方案和项目现状，设计本项目完整的技术架构方案设计，并基于模板生成技术方案文档
步骤四：让用户确认结果

强制要求：
1. 必须严格执行“先文档、后编码”的要求，方案未确定之前不能编码
2. 技术方案设计，必须严格按照项目的规则和要求上下文实现
