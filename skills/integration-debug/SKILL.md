# Integration Debug

## 目标

联调不是单点修 bug，而是把前端页面、前端请求层、后端接口、后端依赖服务甚至更下游能力串起来做真实调试，确认“一个用户动作”从触发到返回、从返回到渲染的整条链路是否成立。

该技能的目标是：

- 先搞清楚联调范围、接口来源和验收口径。
- 先熟悉需求来源，明确本次联调是在满足什么业务目标。
- 在正式调试前建立对项目全局链路、业务数据来源和数据流转的整体认知。
- 基于接口文档、在线文档或后端项目代码，建立当前可用的接口清单。
- 在尽量不破坏现有前端 UI 样式和业务逻辑的前提下完成联调。
- 对异常数据、空数据、脏数据和接口波动做兼容性判断，避免页面因为取数异常直接崩溃。
- 联调结束前做整体流程测试，而不是只盯住单个接口请求。
- 把阻塞问题、证据、推测和待确认项明确收敛，而不是在调试过程中无边界发散。

## 什么时候使用

- 用户要求做前后端联调、接口联调、功能串联调试、验收前联调。
- 已有前端页面或功能，需要对接真实后端接口或下游服务。
- 接口文档不完整，但用户能提供后端项目位置，希望通过扫描代码补齐接口清单。
- 需要核对前端请求参数、后端响应结构、字段绑定关系和异常链路。

## 什么时候不必使用

- 只是解释某个局部函数、组件或接口定义，不涉及真实链路调试。
- 只是输出技术方案，还没有进入真实联调或验收阶段。
- 只是纯后端接口开发或纯前端页面开发，没有跨边界串联验证需求。

## 核心原则

- 联调是“前端到后端到下游”的串联调试，不是局部猜测式排查。
- 联调前必须先理解需求；需求既可以来自用户描述，也可以来自项目中的 PRD、README、接口文档、验收文档和设计说明。
- 调试前必须建立全局意识：当前项目怎么跑起来、核心业务链路是什么、业务数据从哪里来、经过哪些转换、最终如何落到页面、状态或副作用上。
- 默认以现有前端 UI 样式和业务逻辑为准，不能为了跑通接口而随意改样式、改交互、改业务判断。
- 默认不生成联调方案文档；只有在用户明确要求沉淀、落盘、输出方案时，才生成联调方案。
- 如果联调发现问题的根因在于接口契约、后端实现、环境配置、权限、网关或更下游服务，要如实定位，不要把问题粗暴折算成前端改造。
- 调试时必须主动考虑异常数据兼容，尤其是取数、解构、映射、聚合和列表渲染逻辑，避免因为字段缺失、类型不稳、空数组、空对象或 null 值导致整个页面崩溃。
- 联调不能停在“接口通了”；必须补一轮整体流程测试，确认业务主链路、关键分支和异常路径仍然正常。
- 如果遇到阻塞问题，不能自行发散成额外需求或大范围重构；必须带着证据、影响面和改动方案向用户确认。
- 对没有证据的部分，允许先做“带标记的推测”，但必须和真实接口返回严格区分，并在后续调用中逐步替换。

需要浏览器证据、日志、网络、性能或页面行为分析时，读取 `references/browser-tooling.md` 选择最合适的浏览器工具。

## 联调前必须拿到的输入

至少要有以下其中一类接口来源：

- 接口文档、在线文档、OpenAPI/Swagger、Apifox、Postman collection。
- 后端项目路径，允许通过扫描路由、控制器、DTO、OpenAPI 描述、RPC/Feign/gRPC 定义等方式生成接口清单。

同时尽量补齐以下上下文：

- 需求说明、用户描述、PRD、设计稿、验收标准或缺陷描述。
- 复现路径、目标页面、目标操作和关键业务场景。
- 前端项目位置和相关页面、组件、请求封装位置。
- 联调环境信息：域名、base URL、鉴权方式、测试账号、请求头、租户或组织标识。
- 如果链路依赖更下游服务，也要明确依赖边界和可观测方式。

## 工作流程

### 1. 先确认联调边界

- 明确本次联调覆盖哪些页面、功能、用户动作和接口。
- 明确需求来自用户口述还是项目文档；如果项目里有需求文档、验收说明、设计稿或故障说明，先读这些文档。
- 明确用户提供的是接口文档、在线文档，还是后端项目路径。
- 如果接口来源缺失，优先向用户索取；如果用户已给后端项目位置，则直接扫描后端项目补齐接口清单。

### 2. 先熟悉需求与验收目标

- 从用户描述、项目文档、接口文档、设计稿和现有页面行为中总结本次联调要达成的业务目标。
- 先写清“这个功能为什么存在”“用户完成什么动作”“页面应该看到什么结果”“哪些情况算联调通过”。
- 如果需求存在歧义、验收口径冲突或业务规则不清，优先向用户确认，而不是直接带着模糊理解往下调。

### 3. 建立项目全局意识

- 先理解项目如何启动、路由如何进入目标页面、页面数据从哪个请求层进入、经过哪些转换、最终如何进入组件渲染和业务判断。
- 明确业务数据来源：本地状态、接口返回、缓存、URL 参数、用户输入、上下文状态、全局 store 或下游聚合结果。
- 明确数据流转链路：请求参数组装 -> 请求发送 -> 响应解包 -> 数据归一化 / 映射 -> 状态存储 -> 页面渲染 -> 用户动作回写。
- 识别全局公共层：请求封装、响应拦截、鉴权、错误处理、数据 normalizer、列表转换器、空态和异常态组件。

### 4. 读取前端与接口上下文

- 读取前端相关页面、组件、路由、请求封装、字段绑定和错误处理代码。
- 读取接口文档；如果没有文档，则扫描后端项目并梳理接口入口、请求方式、路径、参数、鉴权、响应结构和下游依赖。
- 只聚焦本次联调相关模块，不做无边界全仓扫描。

### 5. 先产出一份当前接口清单

接口清单至少应覆盖：

- 接口名称
- 请求方法
- 请求路径
- 前端触发入口
- 请求参数与关键 headers
- 鉴权 / 租户 / 环境依赖
- 主要响应字段
- 异常响应或错误码
- 是否依赖更下游服务
- 当前状态：已确认 / 待确认 / 可调用 / 不可调用

如果文档和代码不一致，要明确标注冲突点，不要静默选边。

### 6. 逐条核对请求与响应映射

- 核对前端实际发起的请求路径、方法、query、body、headers、鉴权信息是否与接口说明一致。
- 核对响应字段是否满足前端当前绑定关系和业务判断。
- 核对加载中、空态、报错、权限不足、接口失败、部分字段缺失等异常路径。
- 先判断是前端使用错误、接口契约不一致、环境问题，还是后端 / 下游服务异常，再进入下一步处理。

### 7. 主动检查异常数据兼容

- 对取数逻辑默认假设存在异常情况：字段缺失、字段为 null、字段类型漂移、空数组、空对象、分页字段缺省、状态枚举未知、嵌套对象不存在。
- 核对页面是否存在直接解构、直接 `.map()`、直接读取深层字段、未经保护的格式化逻辑或未经兜底的渲染分支。
- 优先在数据归一化、响应映射、页面边界判断或异常兜底层做兼容，避免把异常数据一路透传到渲染层导致整页崩溃。
- 兼容处理的目标是“页面不崩、状态可解释、业务逻辑尽量稳定”，不是掩盖真实问题；如果兼容会影响业务语义，要向用户说明取舍。

### 8. 参数和响应不清晰时的处理规则

如果接口请求可以模拟访问：

- 优先自行构造最小可行请求，获取真实响应。
- 根据真实响应结构回填字段绑定和联调判断。
- 记录样例请求条件，例如 token、环境、必填参数和依赖数据。

如果当前不能模拟访问：

- 允许先基于代码、文档、命名和上下文推测一份“临时模拟数据结构”。
- 必须明确标记这是推测结构，不能伪装成真实接口结果。
- 绑定字段时优先做最小闭环，后续在联调过程中用真实响应逐步替换这些推测字段。

### 9. 遇到阻塞问题时的处理方式

阻塞问题包括但不限于：

- 接口文档缺失或文档与实际不一致。
- 当前环境无法访问接口或鉴权信息不足。
- 后端未实现、返回结构不稳定、字段语义不清。
- 前端现有业务逻辑无法承接真实返回，但改动会影响既有行为。
- 问题根因位于下游服务，不是当前前后端仓库能单独解决。

遇到阻塞时必须输出：

- 问题现象
- 复现条件
- 已有证据
- 初步根因判断
- 可选改动方案
- 对现有 UI、业务逻辑、接口契约和上下游的影响

然后停下来与用户确认，不要自行扩大改动范围。

### 10. 做整体流程测试

- 在局部接口打通后，按真实用户流程回放主链路，而不是只确认单个接口返回正常。
- 至少覆盖：主 happy path、关键前置条件、关键异常路径、接口失败路径、空数据路径和回退路径。
- 如果功能跨多个页面、弹窗、步骤或状态切换，要确认前一步输出的数据能正确进入后一步，而不是只检查当前页。
- 联调测试必须输出明确结论：哪些流程通过、哪些失败、哪些阻塞、哪些尚未验证。
- 需要页面证据时，优先使用浏览器工具补充截图、网络、控制台、性能 trace 或 DOM 观察结果。

### 11. 输出联调结果

联调阶段输出应至少包含：

- 需求来源与本次联调目标
- 本次联调覆盖范围
- 项目全局链路与核心数据流总结
- 读取了哪些前端、文档或后端项目上下文
- 当前接口清单
- 已确认通过的链路
- 仍未打通的链路
- 异常数据兼容处理结论
- 整体流程测试结论
- 阻塞问题与待确认项
- 哪些字段或结构仍然是推测值
- 下一步最小行动建议

默认在对话里给出这些结果即可，不需要自动落盘成联调方案文档。

### 12. 用户要求时再生成联调方案

只有当用户明确要求“生成联调方案”“输出调试方案”“落盘 debug 文档”之类的结果时，才生成方案文档。

生成时必须遵循：

- 使用 `references/template.md` 作为模板骨架。
- 输出目录固定为 `.tmp/docs/debug/`。
- 文件名固定为 `YYYY-MM-DD-[中文名称]-debug.md`。
- `[中文名称]` 必须直接来自本次联调功能语义，使用简洁中文，不要自行改成英文、拼音或泛化标题。

联调方案内容必须重点说明：

- 需求来源与调试目标
- 项目全局链路和业务数据来源
- 哪个功能在联调
- 该功能涉及哪些接口
- 请求参数如何组装
- 响应内容如何处理、绑定、兜底和报错
- 异常数据如何兼容
- 整体流程如何测试

如果用户没有要求联调方案，不要主动落盘，只保留联调过程结论和阻塞项。

## 禁止事项

- 不要为了“先跑通”而随意改现有 UI 样式。
- 不要为了“先跑通”而擅自改前端核心业务逻辑、状态判断或交互语义。
- 不要在没有文档、没有代码、没有实测结果时，把推测结构当成真实接口。
- 不要遇到阻塞就自行扩展需求、重构页面、重写接口层或改动多个系统。
- 不要跳过接口清单整理，直接进入零散试错。
- 不要跳过需求理解和全局链路分析，直接从某一个请求或某一个报错开始盲调。
- 不要只测通一个接口就宣布联调完成，必须补整体流程测试。
- 不要在用户没有要求的情况下，默认生成或落盘联调方案文档。

## 自检清单

- 是否已经明确本次联调覆盖的页面、链路和接口来源？
- 是否已经确认需求来自用户描述还是项目文档，并总结出联调目标与验收口径？
- 是否已经梳理出项目全局业务链路、数据来源和数据流转过程？
- 是否已经拿到接口文档、在线文档，或者后端项目路径中的至少一种？
- 如果没有接口文档，是否已经通过后端项目扫描补齐接口清单？
- 是否始终避免了随意改 UI 样式和前端既有业务逻辑？
- 是否已经检查异常数据、空数据和脏数据对取数与渲染逻辑的影响，避免整页崩溃？
- 接口参数和响应不清晰时，是否优先尝试真实调用获取结构？
- 无法真实调用时，是否把模拟结构明确标记为推测，而不是伪装成真实返回？
- 是否已经补做整体流程测试，并明确哪些流程通过、失败、阻塞或未验证？
- 遇到阻塞时，是否带着证据和改动方案向用户确认，而不是自行发散？
- 如果用户要求输出联调方案，是否使用了 `references/template.md` 并落盘到 `.tmp/docs/debug/YYYY-MM-DD-[中文名称]-debug.md`？

## Agent Notes

- Generic behavior: 先理解需求，再建立项目全局链路认知，随后读取前端与接口上下文、整理接口清单、核对请求与响应、检查异常数据兼容、补整体流程测试，最后输出阻塞项和结论；只有用户明确要求时才额外生成联调方案文档。
- Codex-specific note: 如果运行环境允许读取在线接口文档、抓请求、执行模拟调用、扫描后端项目，或使用浏览器与 DevTools 工具获取页面证据，可以用这些能力提高准确性；但这些工具只是辅助，不能替代“先理解需求和建立接口事实”的流程。
- Fallback behavior: 如果没有在线文档、没有请求工具、也不能直接访问接口，就退回到阅读前端代码和后端项目源码，输出带标记的推测结构与待确认项，等待后续真实联调逐步替换。
