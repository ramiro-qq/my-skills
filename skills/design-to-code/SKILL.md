# Design to Code

## 目标

这个技能用于解决“只有设计稿、截图或原型导出，但需要落成可维护前端代码”的问题。它强调先识别输入类型、归一化设计结构、读取宿主项目规范与可复用组件，再生成高还原实现，并通过截图比对做约束式修复。

它不是“看到图就直接糊页面”的技能，而是一套从输入识别、设计抽象、项目约束对齐、代码生成、视觉验收到迭代修复的完整工作流。对于复杂页面，必须优先保证结构可维护、样式可追踪、修复可迭代，而不是一次性生成一坨不可读代码。

## Quick Start

默认执行顺序：

1. 识别输入来源：Figma、截图、Pencil 或批量任务。
2. 收集设计源、资产和目标页面信息，明确目标 frame/page。
3. 读取宿主项目规范：README、约束文档、目录结构、样式入口、路由和现有页面。
4. 扫描可复用组件、布局壳层、设计 token 和共享 UI。
5. 将设计源归一化为共享 Design IR，而不是直接从原始输入生成代码。
6. 基于项目技术栈和现有规范生成可维护前端实现。
7. 启动本地预览并在目标视口下截图。
8. 将实现结果与设计源做视觉比对，产出可量化的视觉相似度分数并定位差异区域。
9. 仅针对不一致区域做约束式修复，避免破坏已正确部分；每一轮修复后都必须重新截图、重新打分并输出当前视觉相似度分数。
10. 只有视觉相似度达到至少 0.95，或已明确记录外部阻塞原因时，才能结束并输出代码、对比结果、报告和执行日志。

## 核心原则

核心原则：

- 不要拿到设计源就直接写代码；先做输入识别、项目扫描和结构归一化。
- 非 trivial 场景必须先建立 Design IR，再进入代码生成。
- 项目本身的工程规范、命名约定、目录结构和共享组件优先级高于设计工具输出习惯。
- 视觉高还原不等于堆绝对定位；优先使用可维护的 flex、grid 和组件化结构。
- 如果项目没有明确指定组件库，默认使用 shadcn；只有在项目已有既定 UI 体系或用户明确要求其他组件库时才偏离。
- 如果项目里已经有按钮、卡片、表单、布局壳层等公共组件，优先复用，而不是复制一套新的。
- 多页面设计源绝不能默认取第一个页面，必须先解析并确认目标 page/frame。
- 视觉校验必须在页面稳定后执行，不能在加载中、骨架屏阶段或布局抖动阶段截图。
- 视觉相似度 0.95 不是建议值，而是默认验收门槛；低于 0.95 时不得宣称任务已完成，除非存在已记录的外部阻塞。
- 每一轮修复完成后都必须重新截图、重新比对，并明确输出当前视觉相似度分数、上一轮分数、主要差异区域和本轮修复内容。
- 上述每轮修复结果必须立即回显在当前对话中，不能只写入执行日志、差异报告或最终总结；如果没有在对话中回显，则该轮修复视为未完成。
- 修复必须是局部、可解释、可追踪的；不要为了修一个角落把整页推倒重来。
- 批量任务要按 job 隔离输入、输出、报告和失败恢复。
- 整个流程要保留执行日志，确保每一步的输入、输出、改向原因和结果都可追踪。

## 入口选择

根据用户诉求选择入口：

1. **先分析再出实现方案**：适合设计源复杂、项目约束多、还不适合立刻开写时。
2. **直接按设计源实现页面**：适合目标页面和宿主项目范围都比较明确时。
3. **对现有实现做视觉校验与修复**：适合页面已有初版，需要比对设计稿补齐差异时。
4. **批量设计转代码**：适合多个页面、多个 frame 或一组批量任务一起处理时。

如果用户没有明确指定，默认先进入“先分析再出实现方案”。

## 路由规则

路由规则：

- 出现“分析”“方案”“先梳理”“先评估”“先看设计结构”等表述时，优先读取 `references/pipeline.md`
- 出现“实现”“开发”“落地”“转代码”“生成页面”“还原设计稿”等表述时，读取 `references/pipeline.md` 并按完整流程执行实现阶段
- 出现“review”“比对截图”“找差异”“修 UI”“视觉校验”“还原度不够”等表述时，优先读取 `references/acceptance-checklist.md` 与 `references/verification-roadmap.md`
- 出现“日志”“记录过程”“沉淀执行记录”等表述时，读取 `references/execution-log-template.md`
- 出现“批量”“多个页面”“多张图”“多 frame” 等表述时，读取 `references/pipeline.md` 中的 batch 执行约束

如果用户描述同时包含实现与校验，仍然先完成项目规范扫描、Design IR 建立和页面生成，再进入截图比对和修复。

## 工作流

工作流：

- 输入识别：确认是 Figma、截图、Pencil 还是 batch，并锁定目标 page/frame
- 项目扫描：读取 README、约束文档、样式入口、路由、页面目录和共享组件
- 设计归一化：将输入统一成 Design IR，并抽取 token 候选
- 代码生成：遵守项目规范，优先复用共享组件；若项目未指定组件库，则默认采用 shadcn，生成可维护前端代码
- 运行验证：本地预览、稳定态截图、视觉比对、差异报告输出，并生成可量化视觉相似度分数
- 约束修复：只修差异区域；每一轮修复后都必须重新截图、重新比对并输出当前视觉相似度分数，直到达到至少 0.95
- 终止判定：只有视觉相似度达到至少 0.95 时才可标记为完成；若达到修复轮次上限仍未达标，则任务状态必须标记为未完成、blocked 或 needs-human-review，并记录阻塞原因，而不是按成功结束
- 批量收敛：对每个 job 独立输出结果，并汇总总报告


## 对话回显协议

只要进入视觉修复循环，每一轮修复完成后都必须立刻在当前对话中输出一段可见的回合报告，不能等到最终总结时再合并带过，也不能仅写入日志文件。

推荐固定格式如下：

```text
[Visual Repair Round N]
Current score: <current_score>
Previous score: <previous_score>
Delta: <score_delta>
Mismatch regions: <region_1>, <region_2>, <region_3>
Changes made: <what_changed_this_round>
Status: <continue | passed | blocked>
```

执行要求：

- 每一轮都要在聊天对话框中即时回显该报告。
- 如果某一轮修复后没有输出这段回合报告，则该轮视为未完成，必须立即补发。
- `Status: continue` 表示分数仍低于 0.95，需要继续修复。
- `Status: passed` 仅在当前视觉相似度分数至少达到 0.95 时可用。
- `Status: blocked` 仅在存在明确外部阻塞、且已说明阻塞原因时可用。

这个技能执行成功时，应满足：

- 已明确输入类型、目标页面或 frame，而不是默认猜测第一个页面。
- 已扫描宿主项目规范、目录结构、样式体系和共享组件。
- 已为非简单场景建立 Design IR，并据此生成代码。
- 已优先复用项目现有组件、布局壳层和 token，而不是盲目重复造轮子。
- 生成结果具备高视觉还原度，同时代码结构清晰、可维护、可运行。
- 已在页面稳定状态下完成截图校验，而不是在加载中采样。
- 已输出差异报告，并在必要时执行局部迭代修复。
- 每一轮修复后都已输出当前视觉相似度分数，并可追踪分数变化趋势。
- 若最终标记为完成，则视觉相似度必须至少达到 0.95。
- 若未达到 0.95，则必须明确标记为 blocked、未完成或 needs-human-review，并记录外部阻塞原因、受影响区域和下一步修复建议。
- 对模糊设计、冲突约束、错误 frame 选择或改向决策做了明确记录。
- 批量任务中每个 job 都有独立输入、输出、日志和结果沉淀。
- 最终交付同时能通过视觉一致性检查和宿主项目工程规范检查。

## Agent Notes

- Generic behavior: 任何 Agent 都应遵循“输入识别 → 项目扫描 → Design IR → 代码生成 → 视觉验证 → 约束修复”的顺序，不要跳步。
- Codex-specific note: 如果运行环境支持浏览器、截图、图像比对、本地搜索或设计工具集成，可以用来提升证据收集和修复效率，但不能跳过项目规范扫描和组件复用判断。
- Fallback behavior: 如果无法直接读取 Figma 或 Pencil 结构化数据，就基于用户提供的截图、导出资产和项目现状做受约束推断，仍需保留 Design IR 思路、视觉验证和日志记录。
