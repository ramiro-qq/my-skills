# Reference: Context Reading

在进入任何设计稿实现工作前，必须先读取以下上下文：

## 1. 需求来源

至少读取以下其中一种：

- 用户描述
- 需求文档 / PRD
- 技术方案
- 验收标准
- 设计稿说明

目标不是复述原文，而是回答：

- 这个页面或功能解决什么业务问题？
- 用户会做什么动作？
- 页面应该呈现什么结果？
- 哪些行为和状态必须成立？

## 2. 项目规则与约束

优先读取：

- `AGENTS.md`
- README
- 项目内规范文档
- 目录约定、组件约定、主题约定

需要特别关注：

- 技术栈与样式体系
- 路由组织方式
- 状态管理方式
- 组件复用约定
- 主题 token / 变量来源
- 是否已有统一 layout shell

## 3. 代码现状

重点读取与本次需求直接相关的：

- 路由入口
- 目标页面和父级布局
- 页面使用的局部组件与公共组件
- hooks、store、接口请求层
- 样式入口、主题变量、设计 token
- 同类页面或相邻模块

不要做无边界全仓扫描。

## 4. 默认忽略范围

除非用户明确要求，否则不要把下面这些目录当作主要分析对象：

- `node_modules`
- `dist`
- `build`
- `.next`
- `.turbo`
- `coverage`
- 临时导出目录、缓存目录、截图产物目录

## 5. 输出要求

在结束上下文读取后，至少应该能说清：

- 需求来源是什么
- 读取了哪些规则文件
- 读取了哪些页面、组件和相关代码
- 当前页面在业务流程中的位置
- 哪些区域属于本次需求范围
- 哪些区域属于共享壳层或既有能力
