# Reference: Implementation Rules

## 1. 新增需求 vs 增量需求

### 新增需求

符合以下情况之一即可判定为新增需求：

- 当前没有对应功能页面
- 当前只有占位页、空壳页、Demo 页
- 现有结构无法承载需求，且替换成本低于修补成本

默认策略：

- 可以新起页面
- 可以完整覆盖占位实现
- 可以在现有路由下重建内容区

### 增量需求

符合以下情况之一即可判定为增量需求：

- 当前已有部分实现
- 当前页面已经承载真实业务
- 只是补齐设计稿、修正布局、增加状态或提升还原度

默认策略：

- 在现有基础上完善
- 保留能复用的结构、状态流和交互
- 谨慎替换已有公共组件和业务逻辑

## 2. 组件复用优先级

优先级从高到低：

1. 已有公共组件
2. 同模块中可抽离的稳定局部组件
3. 本次新增的页面内私有组件

不要做的事：

- 发现相似组件就立刻大重构
- 为了追求“纯净结构”重写稳定代码
- 明明项目已有组件，却重新造一个相似轮子

## 3. 什么时候应该提公共组件

满足以下大部分条件时，可以考虑提炼：

- 同类结构已在多个页面重复出现
- 视觉和交互已经稳定
- 命名和职责边界比较清晰
- 抽离后不会显著增加当前任务风险

如果只是单次出现，或者还在强业务上下文里，就先保留局部组件。

## 4. 改动边界规则

默认只改这些内容：

- 本次需求相关页面
- 本次需求直接依赖的局部组件
- 为满足复用而做的最小必要公共组件抽取
- 本次改造确实需要的主题变量或样式 token

默认不动这些内容：

- 无关页面
- 共享导航壳层
- 与当前需求无关的全局主题调整
- 为“顺手优化”而产生的大范围重构

## 5. 实现前检查项

编码前必须完成：

- 需求理解
- 业务链路理解
- 项目规则读取
- 设计稿映射
- 需求类型判定
- 组件复用扫描
- 主题和布局分析

## 6. 实现后验收项

至少检查：

- 与设计稿的结构和视觉层级是否一致
- 关键间距、尺寸节奏和组件状态是否合理
- 业务交互和数据状态是否仍然正确
- 空态、加载态、错误态是否补齐
- 是否误改了 header、sidebar 或其他共享壳层
- 是否引入了重复组件或不必要的样式分叉
