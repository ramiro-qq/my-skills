# Implement UI From Design

## 目标

这个技能用于解决“有设计稿，但直接依赖 Figma、Pencil 等 MCP 输出的页面还原度低”的问题。它强调先理解需求、业务逻辑、项目规则和现有代码，再把设计稿映射到真实页面、组件和状态流，最终输出高还原、高一致性、可维护的前端实现。

它不是“看到设计稿就开写”的技能，而是一个从需求理解、设计映射、范围识别、组件复用到最终实现与验收的完整工作流。设计稿中的缺失部分可以推断，但推断必须严格受现有主题、布局、组件语言和业务上下文约束，不能自由发散。

## Quick Start

默认执行顺序：

1. 读取需求来源：用户描述、需求文档、技术方案、设计稿说明。
2. 读取项目规则：优先读取 `AGENTS.md`，其次是 README、约束文档、目录约定。
3. 读取现状代码：路由、页面、组件、接口、状态管理、主题变量和样式入口。
4. 对照设计稿建立映射：明确哪些页面、模块、组件、状态需要实现，哪些是项目现有壳层能力无需改动。
5. 判定需求类型：新增需求或增量需求。
6. 扫描可复用组件和可提炼的公共组件，避免重复造轮子。
7. 先分析设计风格、布局、主题变量和组件体系，再进入编码。
8. 完成实现后，对照设计稿和业务链路做验收，而不是只看静态页面。

## 核心原则

核心原则：

- 不要一开始就写代码；先读需求、业务、项目规则和现状代码。
- 设计稿只是目标表现，真正的实现边界必须结合业务上下文、已有布局和现有组件来判断。
- 对缺失的设计稿内容，只能做“受约束的推断”，必须与已有主题风格、组件逻辑和交互语言保持一致。
- 先判断是新增需求还是增量需求，再决定实现策略。
- 新增需求可以新起页面或完整覆盖占位页；增量需求必须基于现有实现做完善和改造。
- 编码前必须盘点主题变量、布局体系、组件设计和复用机会。
- 如果项目已有公共壳层，例如 header、sidebar、layout、page shell，且用户未明确要求改动，则默认不动这些区域，只改造目标内容区。
- 对于项目中已有但尚未抽公共组件的实现，如果它们已形成稳定复用模式，应按项目约定提炼为公共组件，而不是继续复制。
- 实现结果既要追求设计还原度，也要保持代码结构、复用边界和业务行为的正确性。

## 入口选择

根据用户诉求选择入口：

1. **先分析再出改造方案**：适合设计稿与代码现状差距较大，或需求边界还不清晰时。
2. **直接按设计稿实现页面**：适合需求、设计稿和目标范围已经比较明确时。
3. **审查现有实现与设计稿差异**：适合已有页面需要补齐还原度、风格一致性或缺失状态时。

如果用户没有明确指定，默认先进入“先分析再出改造方案”。

## 路由规则

路由规则：

- 出现“分析”“方案”“先看需求”“先梳理”“先评估”等表述时，读取 `workflows/analyze-and-plan.md`
- 出现“实现”“开发”“落地”“编码”“改页面”“还原设计稿”等表述时，读取 `workflows/implement-from-design.md`
- 出现“review”“对比设计稿”“找差异”“补齐还原度”“检查 UI” 等表述时，读取 `workflows/review-design-gap.md`

如果用户描述同时包含实现与分析，仍然先执行 `workflows/implement-from-design.md` 中的分析前置步骤，再进入编码。

## 工作流

工作流：

- `workflows/analyze-and-plan.md`：只做需求理解、设计映射、范围确认和改造方案收敛
- `workflows/implement-from-design.md`：完成分析后进入实际页面实现与验收
- `workflows/review-design-gap.md`：审查现有页面与设计稿差异并给出修正清单

## 成功标准

这个技能执行成功时，应满足：

- 已读取需求来源、项目规则和相关代码现状，而不是只盯着设计稿。
- 已明确页面或功能的业务目标、作用范围和实现边界。
- 已建立设计稿与代码模块、页面区域、组件和状态的对应关系。
- 已区分新增需求或增量需求，并据此选择了合适的实现方式。
- 已识别并复用了项目中的公共组件、布局壳层和主题能力。
- 对设计稿缺失部分给出了受约束、风格一致的推断，而非自由发挥。
- 实现或方案只覆盖本次需求范围，不误改项目已有壳层与无关区域。
- 最终输出能同时经受设计还原度检查和业务验收检查。

## Agent Notes

- Generic behavior: 任何 Agent 都应遵守“先需求和现状分析，后设计映射，再实现”的顺序，避免直接从设计稿生成低还原页面。
- Codex-specific note: 如果运行环境支持浏览器、设计工具 MCP、截图或本地代码搜索，可用来收集设计证据与现状上下文，但不能跳过对业务逻辑、规则文件和现有组件的读取。
- Fallback behavior: 如果没有设计工具 MCP 或无法直接读取设计稿，就基于用户提供的截图、说明文档和现有代码进行映射与推断，仍需保持主题一致和边界收敛。
